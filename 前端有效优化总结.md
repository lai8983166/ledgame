# 前端有效优化总结

本文总结 `ledGame` 前端在当前阶段已经形成价值的优化方向，重点说明这些优化解决了什么问题，以及对后续交付和扩展带来的作用。

## 1. 前端页面结构优化

前端从早期功能集中在 `App.vue` 的形态，逐步拆分为清晰的页面结构。`App.vue` 主要负责应用壳层、导航切换、Electron IPC 状态接入和全局运行状态维护；具体业务界面下沉到 `src/views/` 中。

目前已经形成的主要页面包括：

- `DemoView`：承载 Demo 调试、运行预览和输入调试相关界面。
- `GameListView`：承载游戏列表入口。
- `SimpleGameEditorView`：承载 Simple 游戏编辑器。
- `MediaLibraryView`：承载媒体库浏览。
- `SpiritLibraryView`：承载精灵库浏览。

这类拆分让页面职责更清楚，后续继续增加系统设置、玩家端、副屏、积分联动等页面时，不需要继续堆叠到主组件中，维护成本更低。

## 2. Electron IPC 边界优化

前端页面不直接关心后端地址、文件系统路径和本地资源访问，而是通过 Electron main/preload 进行受限封装。

主要优化点包括：

- 媒体库通过受限 IPC 访问本地 `media/` 目录，避免 renderer 直接读取文件系统。
- 精灵库通过 Electron IPC 访问后端接口，保持与现有请求模式一致。
- 游戏启动、停止、输入、状态获取等能力统一通过 Electron 层封装。
- 运行状态通过 main 进程广播给多个 renderer 窗口，解决调试窗口状态滞后的问题。

这套边界让前端页面更像纯 UI 层，安全性、可维护性和打包后的适配能力都更好。

## 3. Simple 编辑器工作区优化

Simple 编辑器从基础表单编辑，升级为独立的可视化工作区。编辑器支持游戏基础信息、关卡、帧、颜色、矩阵对象和运行操作集中编辑。

有效优化包括：

- 将 Simple 编辑器独立为完整工作区，避免与游戏列表、Demo 页面混杂。
- 将布局调整为基础信息、帧/矩阵编辑、对象编辑三块区域。
- 将帧序列移动到顶部主区域，给长帧列表更多横向空间。
- 将保存、校验、启动、停止、打开预览等操作重新归类，减少操作分散。
- 压缩关卡、帧、状态栏和输入区域的占用，让 RGB 矩阵获得更多有效显示空间。
- 控制页面级滚动，把必要滚动收敛到矩阵、对象列表或局部面板中。

这些优化让编辑器更接近实际生产工具，而不是临时调试页面。

## 4. 矩阵对象编辑能力优化

矩阵编辑从单点绘制升级为对象化编辑。前端可以表达单格对象、多格对象、对象层级和对象重叠关系。

主要能力包括：

- 点击空白格创建对象。
- 拖拽框选创建多格对象。
- 点击对象占用格选中对象。
- 支持对象移动、旋转、删除、改色、复制和修改基准点。
- 支持对象层级调整，重叠格默认以最上层对象为准。
- 支持将对象层级应用到其他帧。
- 支持当前帧重复次数批量应用到当前关卡所有帧。

这类优化让编辑器可以承载真实游戏内容制作，不再局限于简单点阵测试。

## 5. 全景编辑体验优化

Simple 编辑器增加了全景编辑模式，用于处理真实矩阵边界外的临时编辑和对象调整。

主要优化包括：

- 在真实 RGB 矩阵四周提供固定扩展区域。
- 真实区域和虚拟区域做明显视觉区分。
- 支持在虚拟区域创建、移动和选择对象。
- 支持全景选择模式，将多个单格对象合并为多格对象。
- 虚拟区域对象不会进入最终保存 payload，避免污染真实运行数据。

全景模式让对象移动、合并、边界调整更自然，尤其适合较复杂的灯板游戏编辑场景。

## 6. Canvas 渲染性能优化

矩阵主编辑区从 DOM button grid 迁移到 Canvas 渲染，显著降低了全景模式和大矩阵下的 DOM 压力。

核心优化包括：

- 用 Canvas 替代大量格子 DOM 节点。
- 通过坐标换算处理点击、拖拽、右键菜单和 hover。
- 将 base canvas 和 overlay canvas 拆分。
- base canvas 只负责格子本体、颜色、真实/虚拟区域和重叠标记。
- overlay canvas 负责 hover、拖拽框、选中对象、合并选择和基准点高亮。
- pointermove 不进入 Vue 响应式链路，避免鼠标移动触发整张矩阵重绘。

这项优化让矩阵编辑区更适合承载全景、大尺寸矩阵和复杂对象交互。

## 7. 帧切换与矩阵缓存优化

为了提升帧序列拖动、切帧和对象添加时的流畅度，编辑器增加了多层缓存和局部绘制策略。

主要优化包括：

- 帧序列拖动采用 preview/commit 两阶段，拖动中只预览，松手后再提交切帧。
- 拖动帧序列时冻结对象列表和对象编辑状态，减少连续刷新。
- 为每个 frame 缓存矩阵绘制模型。
- 为每个 frame 缓存对象占用索引，点击命中、重叠判断和矩阵绘制复用同一份数据。
- 使用后台预热队列，优先预热当前帧附近的矩阵缓存。
- 添加对象时支持 patch 绘制，只重绘新增对象格子和旧选中对象格子。
- 选中、高亮、合并选择等纯视觉变化只更新 overlay，不打掉 base 缓存。

这些优化降低了重复计算和重复绘制，提升了编辑器在多帧、多对象场景下的响应速度。

## 8. 窗口适配与缩放优化

Simple 编辑器采用整体 fit viewport 缩放策略，在窗口较小、系统显示缩放较高或任务栏压缩可用区域时，仍保证编辑器完整显示。

主要优化包括：

- 为编辑器增加 viewport、shell、content 三层缩放容器。
- 使用 `ResizeObserver` 和窗口 resize 测量可用区域。
- 按宽高比例计算整体缩放值，最大不超过 1。
- 小屏时整体等比缩放，不再依赖页面滚动找回被裁掉内容。
- 矩阵滚轮缩放、全景切换、帧切换与整页 fit 缩放解耦，避免页面横向漂移。
- Electron 主窗口使用工作区尺寸计算、内容区尺寸和最大化启动，减少 Windows 边框和系统缩放带来的裁切。

这类优化提升了目标机器上的可用性，尤其适合分辨率较低或 Windows 显示比例较高的现场环境。

## 9. 运行预览与调试体验优化

前端增加了面向运行时的预览和调试能力，让编辑、运行、输入验证形成闭环。

主要优化包括：

- Debug panel 可以显示 LED frame、运行状态、frame 尺寸和 hover 坐标。
- Simple 游戏启动后可以打开或聚焦运行预览窗口。
- 运行预览矩阵支持点击输入，并将输入发送给后端 game input 接口。
- 调试窗口可以展示玩法状态，包括分数、生命值、阶段和最近输入结果。
- 运行状态通过 WebSocket 推送接入，减少前端主动轮询。
- 多窗口通过 Electron main 统一广播状态，避免主窗口和调试窗口状态不一致。

这些优化让前端不仅能编辑游戏，还能直接观察游戏运行效果和输入反馈。

## 10. 媒体库与精灵库浏览优化

媒体库和精灵库让游戏编辑资源开始沉淀为可浏览、可复用的系统模块。

媒体库优化包括：

- 展示本地媒体目录树。
- 支持文件夹展开、收起和文件选中。
- 支持图片、GIF、视频预览。
- 对不可预览文件、空目录和读取失败提供明确状态。
- 通过受限协议生成预览地址，避免路径越界。

精灵库优化包括：

- 新增独立精灵库 tab。
- 支持从后端读取精灵列表。
- 支持加载、空状态、错误状态和刷新。
- 支持左侧列表选择、右侧点阵预览和核心字段展示。

这两类资源库为后续素材调用、组件复用和游戏快速制作打下基础。

## 11. 桌面打包与交付体验优化

前端侧开始从开发运行形态转向可交付桌面应用形态。

主要优化包括：

- 接入 Electron Builder，生成 Windows x64 免安装目录和 zip 产物。
- 打包产物内置前端、后端 jar、Java Runtime、初始数据库和 media 资源。
- 打包态由 Electron main 启动内置后端，并管理后端生命周期。
- 首次启动复制 seed 数据库到用户目录，后续启动不覆盖已有用户数据。
- 后端端口支持自动选择，避免目标机器端口冲突。
- 打包态等待后端可用后再打开或请求，失败时提供明确错误。
- 开发模式继续使用外部后端，不破坏原有调试流程。
- portable 产物过滤 Markdown 文档，避免把开发说明和 OpenSpec 文档混入运行包。

这类优化让项目更接近可复制、可部署、可现场测试的桌面软件。

## 12. 综合收益

这些前端优化集中解决了四类问题：

1. **结构更清晰**：页面拆分、IPC 边界和资源库模块化，让后续功能更容易继续扩展。
2. **编辑器更可用**：Simple 编辑器具备了对象化、全景、层级、缓存和缩放能力，更接近真实生产工具。
3. **运行闭环更完整**：编辑、保存、启动、预览、输入、状态推送已经串成前端闭环。
4. **交付更现实**：Electron 打包和内置后端启动能力，让项目可以脱离开发环境在目标机器上运行。

总体来看，前端已经从“Demo 调试界面”推进到“可视化编辑工具 + 资源管理 + 运行预览 + 桌面交付壳层”的阶段，为后续系统设置、玩家端、副屏、积分联动和硬件现场调试继续扩展提供了较稳的基础。

## 13. 近期 Simple 编辑器性能优化链路

最近一轮优化主要围绕 Simple 编辑器中的 RGB 矩阵、全景编辑、帧序列拖动和对象选择展开。优化目标不是单纯追求某一次操作变快，而是把“频繁变化的临时状态”和“真实矩阵数据”拆开，让编辑器少做无关计算。

### 13.1 从 DOM grid 迁移到 Canvas

早期矩阵编辑区如果用大量 DOM button 表示每个格子，格子数量一多，浏览器需要维护大量节点、样式、hover 状态和事件命中。进入全景模式后，矩阵范围还会扩大，DOM 压力更明显。

迁移到 Canvas 后，矩阵变成少量 canvas 节点：

```text
原来:
每个格子 = 一个 DOM 节点 + 样式 + 事件状态

现在:
全部格子 = 一张或两张 Canvas 画布
```

收益是 DOM 数量大幅减少，浏览器布局和样式计算压力下降。矩阵点击、拖拽和右键菜单通过 canvas 坐标反推格子，而不是依赖每个格子的 DOM 事件。

### 13.2 拆分 base canvas 和 overlay canvas

Canvas 化后，如果鼠标 hover 每移动一个格子就重画整张矩阵，仍然会卡。于是矩阵被拆成两层：

- `base canvas`：绘制稳定内容，比如真实格子、虚拟格子、对象颜色、重叠标记。
- `overlay canvas`：绘制临时内容，比如 hover、拖拽框、选中框、合并选择框、基准点高亮。

```text
base canvas    = 较少变化，尽量不重画
overlay canvas = 高频变化，只画少量临时图形
```

这样鼠标移动时只清理和重画 overlay，不再重画所有格子。全景模式下 hover 跟随明显更顺。

### 13.3 hover 状态脱离 Vue 响应式链路

hover 格子是非常高频的临时状态，如果放进 Vue `ref`，每次 pointermove 都可能触发组件响应式更新。优化后 hover、拖拽起点、拖拽终点改为组件内部普通变量。

效果是：

- pointermove 不再推动整棵 Vue 组件更新。
- hover 变化只触发 overlay canvas 的 `requestAnimationFrame` 绘制。
- 鼠标移动频率再高，也不会把对象列表、表单、外层布局一起拖进更新链路。

### 13.4 帧拖动改为 preview / commit 两阶段

拖动帧序列时，用户需要看到矩阵预览，但不一定需要右侧对象列表、对象编辑表单每一个 pointermove 都跟着刷新。

现在的逻辑是：

```text
拖动中:
previewFrameIndex 变化
只刷新矩阵预览和帧 marker

松手后:
activeFrameIndex 提交
对象列表和对象编辑状态再切到目标帧
```

这让拖动帧序列时减少了大量无关 UI 刷新。矩阵仍然跟手，对象区保持稳定，松手后再一次性同步。

### 13.5 矩阵绘制模型按 frame 缓存

矩阵渲染前需要把一帧里的对象展开成每个格子的显示模型。这个过程包括对象点位展开、重叠判断、颜色计算和标题生成。

优化后，每个 frame 会缓存自己的 `matrixCells`：

```text
frame.matrix
  -> matrixCells
  -> Canvas 绘制
```

缓存 key 主要由以下信息组成：

- frame 内部版本号 `__matrixVersion`
- 矩阵范围和尺寸
- 颜色配置

只要 frame 数据没变，切回这帧时就可以复用已有绘制模型，不必重新展开整张矩阵。

### 13.6 当前附近帧后台预热

仅靠“访问后缓存”仍有问题：第一次拖到某帧时，还是会临时计算。于是增加了后台预热。

预热顺序以当前帧为中心向两侧扩散：

```text
current
current + 1
current - 1
current + 2
current - 2
...
```

预热使用 `requestIdleCallback`，浏览器空闲时分批计算；不支持时退化为 `requestAnimationFrame` 小批次执行。这样常用的当前附近帧通常已经提前准备好，而预热本身也不会一次性阻塞界面。

### 13.7 base 数据和视觉高亮分离

一开始 `matrixCells` 里包含了选中对象、合并选择、基准点高亮等视觉状态。问题是这些状态变化很频繁，但它们并不改变“格子里有什么对象”。

优化后拆成两类：

```text
matrixCells:
格子颜色、真实/虚拟区域、对象占用、重叠层数

matrixOverlayHighlights:
选中对象、合并选择、基准点高亮
```

因此点击对象、切换合并选择、进入基准编辑时，不会打掉 `matrixCells` 缓存，也不会触发 base canvas 重绘，只需要更新 overlay canvas。

### 13.8 按 frame 缓存对象命中索引

对象命中、重叠判断和矩阵绘制都会用到同一种结构：

```text
"x:y" -> occupants[]
```

如果每个入口都从 `frame.matrix` 重新展开，会造成重复计算。现在为每个 frame 增加了 `frameOccupancyCache`：

```text
frame.matrix
  -> frameOccupancyCache
  -> 点击命中 / 重叠判断 / matrixCells 构造复用
```

缓存同样跟随 `__matrixVersion` 失效。对象移动、删除、旋转、改色、复制等会修改 frame matrix 的操作，会清掉对应 frame 的命中索引；纯视觉状态不会影响它。

### 13.9 局部 patch 绘制

添加对象时，不一定需要重画整张 base canvas。编辑器可以把新增对象涉及的格子作为 patch 传给 Canvas，只清理并重画这些格子附近区域。

目前已经用于对象添加场景。这个方向后续还可以扩展到移动、删除、改色、旋转等操作：

```text
移动对象 = patch 旧位置 + 新位置
删除对象 = patch 被删除对象位置
改色对象 = patch 当前对象位置
旋转对象 = patch 旧形状 + 新形状
```

### 13.10 当前优化后的原则

这一轮优化形成了几条比较稳定的前端原则：

1. 高频状态不进 Vue 响应式链路，能用组件内部变量就不用全局响应式。
2. 临时视觉状态画在 overlay，不影响 base 缓存。
3. frame 数据不变时，绘制模型和命中索引都复用缓存。
4. 拖动预览和最终提交分开，预览阶段只刷新必要区域。
5. 预热只做当前附近和小批次，不把优化变成新的卡顿源。

这些原则比某个具体缓存更重要。后续继续增加复杂对象、更多帧、更大矩阵或更多编辑工具时，都应该优先检查：这个操作到底改变了真实数据，还是只改变了视觉状态；它应该触发 base 重绘，还是只触发 overlay 更新。
